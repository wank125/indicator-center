## 二、P0~P3 分级标准

### 分级原则

| 优先级 | 定位 | 交付标准 |
|--------|------|----------|
| **P0** | 核心可用 | 跑通"数据抽取→公式计算→日聚合→前端看板"全链路，覆盖电量/电价/电费三大类 |
| **P1** | 功能完整 | 扩展至偏差/差价合约/边际贡献/负荷率，前端4页全部可用 |
| **P2** | 深度分析 | 集团聚合 + 同环比 + 排名，支持多省网数据对比 |
| **P3** | 生态扩展 | 绿色权益 + 外部系统对接 + 高级分析 + 性能优化 |

### 依赖关系

```
P0 ──→ P1 ──→ P2 ──→ P3
 │       │       │
 │       │       └── 需要P1的完整指标数据做同环比
 │       └── 需要P0的计算引擎做更深层计算
 └── 全链路基础，后续所有阶段依赖
```

#### Layer 1: 基础计算 (9个)

| 指标编码 | 公式 | 说明 |

| MLT_CONTRACT_PRICE | `{MLT_CONTRACT_FEE} / {MLT_CONTRACT_ENERGY}` | 合约均价 |
| DAM_AVG_PRICE | `AGG_AVG({DAM_CLEARING_PRICE})` | 日前均价 |
| RTM_AVG_PRICE | `AGG_AVG({RTM_CLEARING_PRICE})` | 实时均价 |
| TOTAL_SPOT_PRICE | `({DAM_CLEARING_ENERGY}*{DAM_CLEARING_PRICE} + {RTM_CLEARING_ENERGY}*{RTM_CLEARING_PRICE}) / ({DAM_CLEARING_ENERGY}+{RTM_CLEARING_ENERGY})` | 现货综合价 |
| SPOT_FULL_FEE | `{DAM_FULL_FEE} + {RTM_FULL_FEE}` | 现货全量电费 |

| DAM_DEV_NEG_ENERGY | `IF({DAM_CLEARING_ENERGY}-{MLT_CONTRACT_ENERGY}<0, {MLT_CONTRACT_ENERGY}-{DAM_CLEARING_ENERGY}, 0)` | 日前负偏差 |
| RTM_DEV_POS_ENERGY | `IF({RTM_CLEARING_ENERGY}-{DAM_CLEARING_ENERGY}>0, ...)` | 实时正偏差 |
| RTM_DEV_NEG_ENERGY | `IF(...)` | 实时负偏差 |
| DAM_DEV_POS_PRICE | `{DAM_CLEARING_PRICE}` | 正偏差电价 |
| DAM_DEV_NEG_PRICE | `{DAM_CLEARING_PRICE}` | 负偏差电价 |
| RTM_DEV_POS_PRICE | `{RTM_CLEARING_PRICE}` | 实时正偏差电价 |
| RTM_DEV_NEG_PRICE | `{RTM_CLEARING_PRICE}` | 实时负偏差电价 |
| CONTRACT_COVERAGE | `{MLT_CONTRACT_ENERGY} / {ONLINE_ENERGY} * 100` | 合约覆盖率 |
| DEVIATION_RATIO | `({RTM_CLEARING_ENERGY}-{DAM_CLEARING_ENERGY})/{DAM_CLEARING_ENERGY}*100` | 偏差率 |

### 3.2 P0 后端任务

#### T-P0-1: 数据抽取服务 (DataExtractService)

**目标**: 从11个业务库动态读取原始指标数据

```
实现要点:
├── 读取 ind_data_source 配置，动态构建 SQL
├── @DS("spot_market") 动态数据源切换
├── 实现 28 个原始指标数据抽取 (Layer 0)
├── 抽取结果写入 ind_data_timeseries (96点分时)
└── 异常处理: 单指标失败不阻塞
```

**关键文件**:
- 修改: `service/DataExtractService.java` (当前空壳)
- 读取: `ind_data_source` 表的 28 条映射记录
- 涉及数据源: spot_market / transaction / settlement / basic_parameter / algorithm / group_data

**验收**: 调用抽取服务后，`ind_data_timeseries` 有 38 个原始指标的 96 点分时数据

#### T-P0-2: 公式引擎集成 (FormulaParseService + FormulaEvalService)

**目标**: 将 FormulaCalcEngine.java 集成到 Spring Service 层

```
实现要点:
├── FormulaParseService — 从 DB 加载公式，校验 + 依赖提取
├── FormulaEvalService — 调用 FormulaParser/FormulaEvaluator
├── 支持 P0 需要的 2 种公式: 四则运算(62个) + 条件分支(38个)
├── 公式中的 {CODE} 替换为 CalcContext 中的实际值
└── 计算结果写入 ind_data_timeseries
```

**关键文件**:
- 修改: `service/FormulaParseService.java` (当前空壳)
- 新建: `service/FormulaEvalService.java`
- 参考: `engine/FormulaCalcEngine.java` (已有完整实现)

**验收**: 输入 `{DAM_CLEARING_ENERGY} - {MLT_CONTRACT_ENERGY}`，给定测试值，能正确计算偏差

#### T-P0-3: DAG 调度 + 逐层计算 (IndicatorCalcService)

**目标**: Layer 0→1→2→3 逐层执行计算

```
实现要点:
├── 从 DB 加载公式，构建依赖图
├── Kahn 算法拓扑排序，按 calc_layer 分层
├── Layer 0: 数据抽取 (38个原始指标)
├── Layer 1: 基础计算 (9个)
├── Layer 2: 偏差计算 (10个)
├── Layer 3: 电费计算 (33个)
├── 计算日志写入 ind_calc_task_log + ind_calc_detail_log
└── 单指标失败不阻塞同层其他指标
```

**关键文件**:
- 修改: `service/IndicatorCalcService.java` (当前仅 triggerCalc 骨架)
- 依赖: T-P0-1 (数据抽取) + T-P0-2 (公式引擎)

**验收**: 手动触发后，90 个 P0 指标全部计算成功

#### T-P0-4: 日/月聚合服务 (AggregateService)

**目标**: 96点分时 → 日汇总 → 月汇总

```
实现要点:
├── AggregateService — 分时数据 → 日汇总 (SUM/MIN/MAX/AVG)
├── 日数据写入 ind_data_daily
├── 月聚合: 日 → 月汇总 + 年累计，写入 ind_data_monthly
├── 聚合触发: 计算完成后自动执行
└── 幂等设计: 同日期重复聚合覆盖旧数据
```

**关键文件**:
- 新建: `service/AggregateService.java`
- 修改: `mapper/IndDataDailyMapper.java` + `IndDataMonthlyMapper.java` (如需自定义SQL)

**验收**: 分时数据存在后，日/月聚合自动产出

#### T-P0-5: 定时调度 (IndicatorCalcJob)

**目标**: 每日凌晨自动执行 T+1 计算

```
执行流程:
1. 创建任务日志 (ind_calc_task_log)
2. 数据抽取 (Layer 0, 38个原始指标)
3. 逐层计算 (Layer 1-3, 52个计算指标)
4. 日/月聚合
5. Redis 缓存刷新
6. 更新任务日志 (成功/失败/耗时)
```

**关键文件**:
- 新建: `job/IndicatorCalcJob.java`
- 配置: `config/ScheduleConfig.java`

**验收**: 凌晨自动执行，任务日志记录成功/失败详情
