# 指标中心 — P0 任务详细设计

> P0 目标：90 个指标 (Layer 0-3) 跑通「数据抽取 → 公式计算 → 日聚合 → 前端看板」全链路。

---

## 任务总览

```
T1 数据抽取 ──→ T2 公式引擎 ──→ T3 DAG调度 ──→ T4 聚合 ──→ T5 缓存 ──→ T6 前端
     │               │              │             │            │
     └── 38个原始指标   └── 52个公式    └── Layer0→3  └── 日/月    └── Redis
```

| 任务 | 优先级 | 依赖 | 预估工时 |
|------|--------|------|----------|
| T1: 数据抽取服务 | 最高 | 无 | 3天 |
| T2: 公式引擎集成 | 最高 | T1 | 3天 |
| T3: DAG调度与计算 | 高 | T1+T2 | 2天 |
| T4: 日/月聚合 | 高 | T3 | 1.5天 |
| T5: Redis缓存 | 中 | T4 | 1天 |
| T6: 查询API补全 | 中 | T5 | 2天 |
| T7: 定时调度 | 中 | T3+T4 | 1天 |
| T8: 前端看板页 | 中 | T6 | 3天 |
| T9: 前端查询页 | 中 | T6 | 3天 |
| T10: 联调验证 | — | 全部 | 2天 |

### 目标
从 6 个业务库的 8 张源表抽取 38 个原始指标，写入 `ind_data_timeseries`。

### 源表映射 (来自 ind_data_source 的 28 条记录)

| 数据源 | 源库 | 源表 | 抽取指标 | 关键字段 |
|--------|------|------|----------|----------|
| 机组参数 | bdss_basic_parameter | bas_unit | RATED_CAPACITY, MIN_TECH_OUTPUT, AUX_RATE | ratedCapacity, minTechOutput, auxRate |
| 变动成本 | bdss_cost_manage | cost_unit_curve | RT_VAR_COST_PRICE | price |
| 合约匹配 | bdss_transaction_manage | trxn_contract_match_month | BASE_CLEARING_ENERGY, MLT_CLEARING_ENERGY, MLT_CONTRACT_PRICE | power, price |
| 日前出清 | bdss_spot_market | spot_ahead_trade | DAM_CLEARING_ENERGY, DAM_CLEARING_PRICE | power, price |
| 实时出清 | bdss_spot_market | spot_real_trade | RTM_CLEARING_ENERGY, RTM_CLEARING_PRICE | power, price |
| 上网电量 | bdss_settlement_manage | sett_power_hour_unit | ONLINE_ENERGY | power |

### DynamicQueryMapper

```java
@Mapper
public interface DynamicQueryMapper {
    @Select("${sql}")
    List<Map<String, Object>> executeQuery(@Param("sql") String sql);
}
```

### 关键逻辑

1. 查询 `ind_data_source` 表获取 28 条映射配置 (status=1, source_type=1)
2. 按 `source_db` 分组，减少数据源切换次数
3. 对每组执行 `@DS(sourceDb)` + `dynamicQuery.executeQuery(sql)`
4. 将结果转为 `IndDataTimeseries` 列表 (96点/天)
5. 批量写入 `ind_data_timeseries` (先删后插，幂等)
6. 单数据源失败记录日志，不阻塞其他

### 验收
- 调用 `extractAll(2026-04-24, 1)` 后
- `ind_data_timeseries` 有 38 个指标 × 96点 = 3,648 条记录
- 每条记录的 `indicator_code`, `trade_date`, `time_point`, `value` 正确

## T2: 公式引擎集成 (FormulaParseService + FormulaEvalService)

### 目标
将 `engine/FormulaCalcEngine.java` 的 21 个类集成到 Spring Service 层，支持 P0 的 52 个计算公式。

### FormulaCalcEngine 现有 API

```java
// 公式解析
FormulaParser.detectType(expr)           → FormulaType
FormulaParser.extractDepCodes(expr)      → Set<String>
FormulaParser.parse(expr)                → List<Token>

// 公式执行
FormulaEvaluator.evaluate(ctx, expr)     → BigDecimal
FormulaEvaluator.evaluateArithmetic(ctx, tokens)  → BigDecimal
FormulaEvaluator.evaluateConditional(ctx, expr)   → BigDecimal

// DAG调度
DAGScheduler.schedule(indicators)        → List<CalcLayer>
CalcExecutor.execute(layers, loader, saver) → CalcResult

// 上下文
CalcContext.put(code, value)
CalcContext.get(code)                    → BigDecimal
```

### P0 公式类型需求

| 类型 | 数量 | 说明 | 示例 |
|------|------|------|------|
| 四则运算 (type=1) | ~35个 | 加减乘除 | `{MLT_CONTRACT_ENERGY} * {MLT_CONTRACT_PRICE}` |
| 条件分支 (type=2) | ~17个 | IF正负判断 | `IF({X}-{Y}>0, {X}-{Y}, 0)` |

### 需新建/修改文件

```
bdss-indicator-biz/src/main/java/com/qctc/bdss/indicator/
├── service/
│   ├── FormulaParseService.java         ← 重写 (当前空壳)
│   └── FormulaEvalService.java          ← 新建
└── engine/                              ← 将 FormulaCalcEngine 拆分为独立类
    ├── FormulaParser.java               ← 从 FormulaCalcEngine 提取
    ├── FormulaEvaluator.java            ← 从 FormulaCalcEngine 提取
    ├── DAGScheduler.java                ← 从 FormulaCalcEngine 提取
    ├── CalcExecutor.java                ← 从 FormulaCalcEngine 提取
    └── CalcContext.java                 ← 从 FormulaCalcEngine 提取
```

### 核心接口设计

```java

     * 校验公式 (语法 + 循环依赖)

     * 执行单个公式
     * @param formula  公式记录

     * @return 计算结果
     */
    public BigDecimal evaluate(CalcFormula formula, CalcContext context);

    /**
     * 批量执行同层公式
     * @param formulas 同层公式列表
     * @param context  共享上下文

```

### 引擎拆分策略

将 `FormulaCalcEngine.java` (1025行) 拆分为 5 个独立文件，保持包路径 `com.qctc.bdss.indicator.engine`：

1. **FormulaParser** — 词法分析 + 公式类型检测 + 依赖提取 (约200行)
2. **FormulaEvaluator** — 四则运算 + 条件分支执行 (约350行)
3. **DAGScheduler** — 拓扑排序 + 分层 (约150行)
4. **CalcExecutor** — 逐层执行编排 (约200行)
5. **CalcContext** — ConcurrentHashMap 上下文 (约50行)

### 验收
- `FormulaParseService.extractDependencies("{DAM_CLEARING_ENERGY} - {MLT_CONTRACT_ENERGY}")` → `["DAM_CLEARING_ENERGY", "MLT_CONTRACT_ENERGY"]`
- `FormulaEvalService.evaluate(formula, ctx)` 给定 DAM=100, MLT=80 → 结果 20
- IF 条件: `IF({X}>0, {X}, 0)` 给定 X=-5 → 结果 0

## T4: 日/月聚合服务 (AggregateService)

### 目标
96点分时数据 → 日汇总 → 月汇总 + 年累计。

### 聚合规则

| 指标类型 | 日聚合方式 | 月聚合方式 |
|----------|-----------|-----------|
| 电量类 (MWh) | SUM | SUM + 年累计 |
| 电价类 (元/MWh) | AVG | 加权平均 |
| 电费类 (元) | SUM | SUM + 年累计 |
| 容量类 (MW) | MAX | MAX |
| 比率类 (%) | AVG | AVG |

### 需新建文件

```
service/AggregateService.java            ← 新建
dto/request/AggregateReq.java            ← 新建
```

### 核心接口设计

```java
@Service
@Slf4j
public class AggregateService {

    /**
     * 日聚合: 96点分时 → 日汇总
     * @param date     交易日期
     * @param tenantId 租户
     * @return 聚合的指标数
     */
    public int aggregateDaily(LocalDate date, Long tenantId);

    /**
     * 月聚合: 日汇总 → 月汇总 + 年累计
     * @param month    月份 yyyy-MM
     * @param tenantId 租户
     */
    public int aggregateMonthly(String month, Long tenantId);
}
```

### 聚合SQL思路

```sql
-- 日聚合
INSERT INTO ind_data_daily (indicator_id, indicator_code, unit_id, trade_date,
    value, min_value, max_value, avg_value, tenant_id)
SELECT indicator_id, indicator_code, unit_id, trade_date,
    SUM(value), MIN(value), MAX(value), AVG(value), tenant_id
FROM ind_data_timeseries
WHERE trade_date = :date AND tenant_id = :tid
GROUP BY indicator_code, unit_id, trade_date, tenant_id
ON CONFLICT (indicator_code, unit_id, trade_date, tenant_id)
DO UPDATE SET value = EXCLUDED.value, min_value = EXCLUDED.min_value, ...
```

### 验收
- 日聚合后: `ind_data_daily` 每个指标×每个机组每天1条
- 月聚合后: `ind_data_monthly` 每个指标×每个机组每月1条，含 year_cumulative
- 重复执行幂等 (ON CONFLICT 覆盖)
